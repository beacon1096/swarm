# ADR talos-ii/0004 — Official Talos image factory only, no custom extensions (talos-ii)

**Scope:** talos-ii only. **talos-i continues to use a self-hosted image factory** because it requires the `util-linux-mountpoint` custom extension to make Harvester CSI work. See [Constitution §IV](../../../.specify/memory/constitution.md#iv-image-factory).
**Status:** accepted
**Date:** 2026-04-27

## Context

Previous talos-ii / talos-i used a self-hosted image factory (`ghcr.io/beacon1096/image-factory/...`) that built custom Talos images including a `util-linux-mountpoint` system extension. The reason: Harvester CSI's `NodeUnstageVolume` ran `nsenter ... mountpoint ...` against the Talos host, and Talos's standard image doesn't ship `mountpoint` — so the call failed with "No such file or directory" until the extension provided it.

Maintaining the self-hosted factory cost (per cluster that uses it):
- A `talos-image-factory` Deployment in some namespace
- An OCIRepository / Helm chart for the factory image
- Custom Spegel mirror entries to route the factory through Zot
- A Talos schematic ID generated by the self-hosted factory, opaque to anyone reading the repo unless documented elsewhere
- Periodic rebuilds when Talos versions or upstream extensions change

## Decision

**For talos-ii (bare-metal MS-01):**

1. Use **`https://factory.talos.dev`** (sidero's official factory) for all images
2. Include only **published `siderolabs/*` extensions** in the schematic
3. **Never** create or include a custom-built extension for talos-ii

If a workaround on talos-ii seems to require a custom extension, the right move is to fix the underlying issue or change architecture, not to ship a workaround in the image. The bare-metal Longhorn architecture removes the original justification.

**For talos-i (Harvester KubeVirt on NEC8):**

talos-i remains free to use a self-hosted image factory (and is expected to). When talos-i is adopted into this repo, its self-hosted factory configuration moves into a clearly cluster-scoped path (e.g. `kubernetes/clusters/talos-i/registry/talos-image-factory/`).

The cluster-scoping matters because:

- talos-ii's tooling never depends on talos-i's factory being up
- Schematic IDs from each factory live in separate sections of [docs/talos-image-factory.md](../../talos-image-factory.md)
- A new operator reading the repo can tell which images came from which path

## Why the previous workaround is no longer needed (for talos-ii)

`util-linux-mountpoint` was specifically for Harvester CSI's `nsenter mountpoint` call. On talos-ii with bare-metal Longhorn:
- Longhorn's CSI driver uses different paths (`mount`, `umount`, `findmnt`)
- The `siderolabs/util-linux-tools` official extension covers what Longhorn actually needs
- Harvester CSI is not present at all

Verified by reading Longhorn's CSI driver code and matching to the `util-linux-tools` extension contents.

talos-i still has Harvester CSI in the loop, so it still needs the custom extension. **That is not a contradiction with this ADR** — this ADR is scoped to talos-ii.

## Alternatives considered

- **Keep the self-hosted factory for talos-ii's "future flexibility"** — but every "future flexibility" we've actually used in the past (`util-linux-mountpoint` workaround) is what this ADR is trying to *avoid*. Bring it back if and when there's a real concrete reason for talos-ii.
- **Use `talosctl gen secret` and roll our own installer** — even more work than self-hosted factory, with no benefit.
- **Make talos-ii also use the self-hosted factory for "consistency"** — that imports talos-i's complexity into talos-ii without any payoff.

## Consequences

Positive:
- One fewer service for talos-ii to depend on
- talos-ii schematic IDs are public and reproducible by anyone with the schematic YAML
- Talos upgrades on talos-ii follow a well-trodden path (`task talos:upgrade-node IP=...` against an official URL)
- The split makes the per-cluster differences explicit — talos-i's custom extension need is no longer hidden inside a "shared factory"

Negative:
- If sidero's factory is down during a talos-ii re-image, we wait. Acceptable for a homelab. Mitigation: cache the installer image to local Zot.
- talos-ii loses the option to ship custom kernel modules quickly. If we ever need this, the recovery path is: revive a self-hosted factory under a new ADR (could even reuse talos-i's, with explicit cluster scoping in the build pipeline).
- Two factory paths to maintain across the repo (talos-ii's official, talos-i's self-hosted). [docs/talos-image-factory.md](../../talos-image-factory.md) is the single source of truth that documents both.

## Convention

Every change to the Talos schematic is recorded in `docs/talos-image-factory.md` in the **same commit** as the change. The image factory ID is opaque hash; without the doc, future-you cannot recover the extension list.
