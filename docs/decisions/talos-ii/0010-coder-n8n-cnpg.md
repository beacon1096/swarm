# ADR talos-ii/0010 — Coder & n8n on CloudNativePG

**Scope:** talos-ii. Both apps are dev-team tooling.
**Status:** accepted
**Date:** 2026-04-28

## Context

Phase 3e of the talos-ii rebuild: port `coder` (workspace manager)
and `n8n` (workflow automation) from swarm-01. Both:

- Are PG-backed (dumps `coder-pg.sql.gz` and `n8n-pg.sql.gz`, both
  PG 18.3).
- Already had bitnami `postgresql` subchart helmreleases on swarm-01,
  which we replace with CNPG `Cluster` CRs per ADR 0007.
- Land in the existing `development` namespace alongside forgejo.

Three sub-decisions to nail down:

1. Whether to wire **n8n SSO via authentik** (swarm-01 had it
   commented as "planned").
2. What to do with the **coder workspace tarball** (`coder-workspace.tar.gz`,
   476 MB — a single user workspace's `/home/coder/`).
3. **Wildcard hostname** routing for coder (`*.code.beaco.works`)
   given Gateway API limits.

## Decisions

### 1. Coder + n8n on CNPG, both pinned to PG 18.3

Same shape as authentik (ADR 0008) — per-app `Cluster` CR with
`postgresql.enabled: false` (or analogous override in the chart's
own external-PG knob). PG 18.3 image to match the dumps.

For coder the chart consumes `CODER_PG_CONNECTION_URL` directly, so
we synthesize a postgres URL into `coder-secret.pg-connection-url`:

```
postgres://coder:<app-pw>@coder-pg-rw/coder?sslmode=disable
```

For n8n the chart takes `DB_POSTGRESDB_*` env vars individually,
`DB_POSTGRESDB_HOST: n8n-pg-rw`, password from `n8n-pg-app` Secret.

### 2. n8n stays on **local auth**, no SSO

Skip the swarm-01 TODO comment about SAML via authentik. Reasons:

- No `n8n` OAuth/SAML provider exists in the restored authentik DB
  (verified: only `coder` / `forgejo` / `matrix` / etc. are
  registered; `n8n` slug is absent).
- n8n's encryption-key carry-over is already a load-bearing rule
  (DB-encrypted credentials); adding an auth-flow change on top
  introduces a second variable for the same migration.
- n8n local users are tiny (single operator) and never were the
  authentication burden the spec was supposed to relieve.

If/when n8n needs SSO, do it as a follow-up ADR with a separate
authentik provisioning step, not bundled with the restore.

### 3. **Skip restoring the coder workspace tarball.**

The 476 MB tarball is a single user's `/home/coder/` filesystem
state from one workspace pod (mostly bun caches and node_modules).
Coder's design separates two state types:

- **Server-side** (workspaces, users, organizations, templates,
  OIDC claims) lives in PG → restored via the `coder-pg.sql.gz` dump.
- **Workspace-side** (`/home/coder/` per pod) is treated as
  ephemeral or template-rebuilt — coder users `coder rebuild` to
  reprovision a workspace from the Terraform template.

Restoring the workspace tarball would require:

- Identifying which workspace it came from (no manifest).
- Provisioning that workspace's PVC.
- Mounting + untarring before the workspace pod starts.

Not worth the operator time for a workspace that's reproducible
from `templates/kubernetes/main.tf`. The tarball stays in
`.private/talos-ii-export-20260426/` for selective restore later.
Documented in
[operations/coder-restore.md](../../operations/coder-restore.md).

### 4. Coder wildcard subdomain routing

Coder generates URLs like `2-fastapi--myws--user.code.beaco.works`
for browser-rendered ports inside running workspaces. Two
HTTPRoutes:

- `code.beaco.works` (apex) — main UI.
- `*.code.beaco.works` (wildcard) — workspace port forwarding.

The Gateway's `https` listener already terminates `*.beaco.works`
via cert-manager DNS-01 wildcard, so no additional cert config
needed. Cloudflare DNS just needs an `A` record on `*.code.beaco.works`
pointing at the same Cloudflare Tunnel.

### 5. Layout

```
kubernetes/apps/development/
├── coder/
│   ├── ks.yaml
│   └── app/
│       ├── kustomization.yaml
│       ├── pg-cluster.yaml
│       ├── pg-secret.sops.yaml
│       ├── ocirepository.yaml
│       ├── helmrelease.yaml
│       ├── secret.sops.yaml
│       └── httproute.yaml
└── n8n/
    ├── ks.yaml
    └── app/
        ├── kustomization.yaml
        ├── pg-cluster.yaml
        ├── pg-secret.sops.yaml
        ├── data-pvc.yaml
        ├── ocirepository.yaml
        ├── helmrelease.yaml
        └── secret.sops.yaml
```

n8n's HTTPRoute is generated by bjw-s app-template's `route:` value
block — same pattern as forgejo / sing-box.

## Consequences

Positive:

- Reduces the bitnami subchart surface area in the cluster (no more
  per-app PG StatefulSets).
- Encryption-key carry-over keeps n8n credentials usable across
  the migration.

Negative / accepted trade-offs:

- coder workspace state lost (selectively, by skipping the tarball).
  Workspace user re-provisions on first login.
- n8n SSO deferred; one more login surface for the operator until
  someone wants to fix it.
- Two more PG clusters to upgrade when PG 19 lands (5 PG clusters
  total: authentik 18, matrix 15, coder 18, n8n 18 + future ones).
